# Pitfalls Research

**Domain:** Emergency blood-donor matching PWA — Myanmar audience, bilingual (EN/MY), FCM + Supabase + React
**Researched:** 2026-06-19
**Confidence:** HIGH (FCM/iOS push, RLS, PostGIS), MEDIUM (Myanmar locale specifics, abuse vectors), HIGH (blood compatibility direction)

---

## Critical Pitfalls

### Pitfall 1: iOS Push Requires Home-Screen Install — In That Order

**What goes wrong:**
Developers wire up FCM token subscription on page load or shortly after registration, then discover no iOS Safari user ever receives a push notification. The subscription call either silently fails or returns a token that FCM never actually delivers to, because the device is not a Home-Screen web app at the moment of subscription.

**Why it happens:**
iOS 16.4+ only grants push to PWAs launched from the Home Screen (display: standalone). A browser tab visit to the same URL cannot subscribe. The sequence matters: install first, then request permission, then subscribe. Developers who test on Android or desktop never hit this because those platforms allow in-tab subscription. The FCM JS SDK accepts the `getToken()` call on iOS Safari in-tab but the resulting token is inert.

**How to avoid:**
- Gate the FCM `getToken()` call behind a check: `window.matchMedia('(display-mode: standalone)').matches`. If false, do not call `getToken()` — instead show the Add-to-Home-Screen prompt UI.
- Show a contextual install banner before any notification permission request. On iOS this must be a custom UI (no `beforeinstallprompt` event on Safari); instruct the user to tap the Share icon then "Add to Home Screen."
- After install confirmation (user re-opens in standalone mode), then and only then trigger the native permission prompt via a user gesture (button tap), not on mount.
- Fallback: for users who do not install, show an in-app "check back here" fallback rather than silently failing.
- The PWA manifest must have `"display": "standalone"` — without it, iOS will not enable push even after install.

**Warning signs:**
- Zero iOS push deliveries in FCM console while Android shows healthy delivery.
- FCM `getToken()` returns a token on iOS in-browser tab — this is the false positive; check `display-mode` before trusting it.
- Users report "I enabled notifications but never got the alert."

**Phase to address:** PWA install + push subscription phase (the phase that wires FCM token registration). Must be designed before writing any subscription code.

### Pitfall 2: Service Worker Scope Collision Between vite-plugin-pwa and firebase-messaging-sw.js

**What goes wrong:**
Two service workers fight over the same `/` scope. The PWA SW (generated by vite-plugin-pwa/Workbox) and Firebase's `firebase-messaging-sw.js` both want to control all routes. The symptom is an infinite reload loop after the app updates, or push messages that never reach the SW handler, or one SW silently overwriting the other on activation.

**Why it happens:**
vite-plugin-pwa registers its generated SW at `/sw.js` with scope `/`. Firebase requires `firebase-messaging-sw.js` also at root scope `/`. When both are registered, the browser allows only one active SW per scope. Whichever registers last wins, and the loser's cached assets or message handler disappear. The `importScripts()` approach also breaks in Vite because `firebase-messaging-sw.js` cannot use `import.meta` (not a module context) and cannot use `importScripts()` if registered as `type: 'module'`.

**How to avoid:**
- Use the `injectManifest` strategy in vite-plugin-pwa and create a single custom service worker file (e.g., `src/sw.ts`) that imports both Workbox precaching logic and `importScripts('https://www.gstatic.com/firebasejs/[version]/firebase-app-compat.js')`.
- Do NOT register `firebase-messaging-sw.js` as a separate file. Merge Firebase Messaging initialization into the single Workbox SW.
- In the Vite config, use a custom plugin to replace `import.meta.env.*` strings with literals in the SW output, since SW files run outside module context.
- Verify with Chrome DevTools → Application → Service Workers that only one SW is registered at root scope.

**Warning signs:**
- Chrome DevTools shows two SWs registered at scope `/` simultaneously.
- App hard-reloads in a loop after a new deploy.
- Background push messages silently drop (no SW `notificationclick` events).
- Console error: `Cannot use 'import.meta' outside a module` in the SW.

**Phase to address:** PWA scaffolding phase. The merged SW architecture must be decided before any Firebase or Workbox code is written, not retrofitted later.

### Pitfall 3: FCM Foreground Messages Do Not Show a System Notification

**What goes wrong:**
When the app is open in the foreground (the PWA is the active window), FCM's background service worker never fires. The `onMessage()` handler in the app receives the payload, but if no code explicitly calls `new Notification(...)` from that handler, the user sees nothing. This breaks the donor experience: a donor who has the app open gets no alert that a matching blood request was just posted.

**Why it happens:**
The FCM JS SDK splits message handling by app state. Background (SW) → browser auto-shows the notification payload. Foreground (app open) → the SDK fires `onMessage()` and suppresses the automatic SW notification to avoid duplicates. Developers assume FCM will always show a notification, regardless of app state.

**How to avoid:**
- In `onMessage()` handler: explicitly construct and display a `new Notification(title, options)` or render an in-app toast/modal that is equally prominent.
- Prefer sending FCM "data-only" messages (no `notification` key in the payload) so the SW always handles display uniformly, regardless of foreground/background state. Handle display entirely in `notificationclick`/`setBackgroundMessageHandler`.
- Test specifically with the app tab active and confirm the donor sees the alert.

**Warning signs:**
- Push notifications work when the app is closed but not when a donor is actively using it.
- `onMessage` fires (visible in console) but no visible notification appears.

**Phase to address:** FCM push integration phase, as part of end-to-end alert testing.

### Pitfall 4: Stale / Expired FCM Tokens Cause Silent Push Failures

**What goes wrong:**
A donor registered months ago and their FCM token is stored in Supabase. The token has since rotated (user reinstalled browser, cleared storage, FCM rotated it server-side, or the token reached 270 days of inactivity). When a blood request fires, the Supabase Edge Function sends to the stale token. FCM returns an error (`registration-token-not-registered` or `invalid-registration`), but if the error is not caught and acted on, the token stays in the database and that donor is permanently invisible to the system — they will never receive alerts.

**Why it happens:**
FCM token rotation is silent on the client. The token changes but `onTokenRefresh` fires only if the app is open. Developers store the initial token at registration and never update it. FCM does not proactively invalidate old tokens in a way the developer sees — the failure only surfaces when a send fails.

**How to avoid:**
- Subscribe to `messaging.onTokenRefresh()` in the app and upsert the new token to Supabase whenever it fires.
- When FCM send returns an error code of `registration-token-not-registered` or `messaging/registration-token-not-registered`, delete that row from the `fcm_tokens` table immediately in the Edge Function error handler.
- Periodically (e.g., a weekly Supabase pg_cron job) mark tokens as stale if they have not been refreshed in 90 days, and exclude stale tokens from alert sends.
- Tokens table: store `last_refreshed_at` and `last_send_attempted_at`.

**Warning signs:**
- FCM console shows a growing gap between "messages sent" and "messages delivered."
- Some donors have never received any alert despite being registered and in-range.
- Edge Function logs show FCM error responses that are not followed by a token deletion.

**Phase to address:** FCM push integration phase; token lifecycle management should be built at the same time as token storage, not added later.

### Pitfall 5: RLS Misconfiguration Leaks Donor Phone Numbers

**What goes wrong:**
The `profiles` or `blood_requests` table contains callback phone numbers. RLS is either disabled (tables created via SQL have RLS off by default in Supabase) or a policy is written that allows any authenticated user to `SELECT *` on any row. Any logged-in user can then query the full table and harvest all phone numbers. With a dummy auth (no real identity verification), "authenticated" is trivially cheap to obtain.

**Why it happens:**
Supabase ships with RLS off by default on new tables created via SQL. The Supabase Dashboard has an "Enable RLS on new tables" toggle that must be turned on project-wide, but it is not on by default. 10.3% of analyzed Supabase apps in a 2025 CVE disclosure (CVE-2025-48757) had at least one table readable by anyone holding the anon key, and 83% of Supabase security incidents involve RLS misconfiguration.

**How to avoid:**
- Enable RLS on every table at creation time: `ALTER TABLE profiles ENABLE ROW LEVEL SECURITY;`
- Phone numbers should never be in a public SELECT-accessible column. The `blood_requests` table exposes a callback number only to donors who have been explicitly matched and flagged as "responding" — enforce this with a join-based RLS policy.
- Write RLS policies that are as narrow as possible: a donor viewing a request should see only the columns necessary for the alert (blood type, distance, request ID), not the raw phone number until they tap "I'm responding."
- Test RLS with the anon key and a non-owner authenticated JWT using `supabase.auth.setSession()` in tests — not with the service_role key which bypasses RLS entirely.
- Never expose the `service_role` key to the browser. For the dummy auth, all browser-side clients must use the `anon` key only.

**Warning signs:**
- Supabase Dashboard → Table Editor shows "RLS disabled" on any table in the public schema.
- A `SELECT * FROM profiles` query with the anon key returns rows.
- Phone number column is visible in Supabase's auto-generated REST API response without authentication.

**Phase to address:** Database schema + RLS phase. RLS policies must be written and tested before any data is stored, not retrofitted after the UI works.

### Pitfall 6: Blood Compatibility Direction Error (Donor→Recipient, Not Recipient→Donor)

**What goes wrong:**
The matching query is written backwards. The code checks "can the requester's blood type donate to the donor's blood type?" instead of "can the donor's blood type donate to the requester's blood type?" This inverts the entire compatibility matrix. For example, a person requesting B− blood would alert O− donors (correct), but the inverted query would alert B− requesters when an O− donor is available (wrong direction — no one is requesting anything here; the query is for who can donate TO the requester).

**Why it happens:**
The ABO+Rh compatibility matrix is directional and its intuitive reading is ambiguous. "O− is compatible with B−" is true in the direction "O− can donate to B−." A developer who reads this as "O− and B− are compatible" and writes `WHERE donor_type = requester_type OR donor_type IN (...)` without carefully tracking which side is the source can invert the logic. The full matrix has 64 cells (8 types × 8 types) and partial implementation is easy — testing with A+ requesters and A+ donors (same-type match) will appear to work while the cross-type logic is backwards.

**How to avoid:**
- Encode the matrix explicitly as: `COMPATIBILITY[donor_type][recipient_type] = true/false`, not as a symmetric set.
- The lookup direction in the matching query is always: `WHERE compatibility_table[donor.blood_type][request.blood_type] = true`.
- Write the full 8×8 matrix as a SQL lookup table or as a constant in a shared utility: `{ 'O-': ['O-','O+','A-','A+','B-','B+','AB-','AB+'], 'O+': ['O+','A+','B+','AB+'], ... }`. The key is donor type; the values are all recipient types that donor can give to.
- Unit test the matrix with known-hard cases: O− donor → all 8 recipient types should match; AB+ donor → only AB+ recipient; AB− recipient → must accept A−, B−, O−, AB−; O+ recipient → must accept O+, O− but not A+, B+.
- Test with a B− blood request and verify O− donors are alerted but not A+ donors.

**Warning signs:**
- O+ requesters receive alerts to O− donors (correct) but also receive alerts from AB+ donors (wrong — AB+ cannot donate to O+).
- Symmetric testing (same blood type requesting and donating) all passes, but cross-type tests are never written.
- The matrix is implemented as a JS/TS Set of pairs without directional labeling.

**Phase to address:** Matching logic phase. Write the matrix as a tested constant in its own file before any SQL query is written. Do not embed it inline in a query.

### Pitfall 7: Dummy Auth Leaking Into Real RLS Identity

**What goes wrong:**
The dummy OTP flow auto-fills an OTP and considers the user "logged in." If this is implemented by calling `supabase.auth.signInWithPassword()` with a fixed/predictable credential, or by using the service_role key to bypass auth entirely in development, the RLS policies never actually enforce identity. In the worst case, the service_role key is used in the browser-side client during development and never removed before demo — now any user's browser has root access to the database.

A subtler version: the dummy auth creates a Supabase Auth user (real JWT) but does not properly set `auth.uid()` to a stable identifier tied to the phone number. Two sessions for the "same" phone number create two distinct `auth.uid()` values, breaking ownership checks in RLS policies (`WHERE user_id = auth.uid()`).

**Why it happens:**
Dummy auth is designed to avoid real SMS infrastructure, so developers reach for shortcuts: hardcoded users, the service_role key for "simplicity," or anonymous sign-in that creates a new identity each session. These work in isolation but corrupt the RLS identity layer.

**How to avoid:**
- Implement dummy auth using `supabase.auth.signInWithOtp()` or `supabase.auth.signUp()` with email derived from the phone number (e.g., `+95{phone}@bloodhelp.local`) — this creates a real Supabase Auth user with a stable `auth.uid()` that survives across sessions.
- Never instantiate a Supabase client with the `service_role` key in any browser-side code, including during development.
- RLS policies must always reference `auth.uid()` — test them with a real user session, not via the Supabase Dashboard (which runs as superuser).
- Keep the dummy-auth adapter behind a clear interface boundary so that if real OTP is added later, only the adapter changes — not the RLS policies or schema.

**Warning signs:**
- `supabase.auth.getUser()` returns `null` but database writes still succeed (means service_role key is in the browser client).
- The same phone number creates a different `user_id` on each login.
- RLS policies that reference `auth.uid()` return zero rows during testing but the developer "fixes" it by removing the policy.

**Phase to address:** Auth phase (first phase). The dummy auth must produce a real, stable Supabase Auth identity — this is a prerequisite for all subsequent RLS work.

### Pitfall 8: Missing GiST Index Makes Radius Queries Sequential Scans

**What goes wrong:**
The donor-matching query uses `ST_DWithin(donor_location, request_location, radius_meters)` without a GiST spatial index on the `donor_location` column. PostgreSQL falls back to a sequential scan of all donors. At 100 donors this is invisible. At 10,000 donors in Yangon this is a 300ms+ query per blood request — and this query fires inside the critical path of the emergency alert.

**Why it happens:**
PostgreSQL's standard B-tree index does not work on geometry types. Developers who `CREATE INDEX idx_donor_location ON profiles (location)` get a B-tree, which PostGIS ignores entirely. The query planner will still produce results — just slowly, with no error.

**How to avoid:**
- `CREATE INDEX idx_donor_location ON profiles USING GIST (location);` — the `USING GIST` is mandatory.
- Run `EXPLAIN ANALYZE` on the matching query before shipping and verify the plan shows `Index Scan using idx_donor_location` not `Seq Scan on profiles`.
- Use `ST_DWithin` not `ST_Distance(...) < radius` — only `ST_DWithin` can use the GiST index.
- Store location as `geography` type (not `geometry`) to use meter-based distance natively without coordinate projection math.

**Warning signs:**
- `EXPLAIN ANALYZE` shows `Seq Scan` on the profiles/donor table.
- Query latency grows linearly as donor count grows (not logarithmically).
- A `CREATE INDEX` was run without `USING GIST`.

**Phase to address:** Database schema phase, when the `profiles` table is created. Add the index in the same migration as the column.

### Pitfall 9: Supabase Edge Function Cold Start Adding Latency to the Emergency Alert Path

**What goes wrong:**
The blood request → donor alert path goes through a Supabase Edge Function (triggered via a database webhook on `blood_requests` insert). If the function has not been invoked recently, it cold-starts before processing. Even with Supabase's 2025 improvement to ~42ms cold starts, cold start + function execution + FCM HTTP call = potentially 500ms–2s of latency before the first push is sent. For an emergency, this matters.

A worse variant: the database webhook uses `pg_net.http_post()` to invoke the Edge Function, but `pg_net` fails silently if the Edge Function URL is wrong, the JWT header is missing, or there is a network error from the database's perspective. The transaction commits successfully, the webhook appears configured, but no push is ever sent.

**How to avoid:**
- Prefer a database webhook (via Supabase Dashboard → Webhooks) over a raw `pg_net` trigger — the webhook UI handles auth headers and retry logic more reliably.
- Monitor the `net._http_response` table in the `net` schema to verify webhook calls are reaching the Edge Function and returning 2xx.
- For latency: keep the Edge Function warm by pinging it on a schedule (pg_cron every 5 minutes) or accept the cold start and set user expectations accordingly.
- Design the Edge Function to be idempotent and fast: it receives the request row, runs the compatibility+distance SQL, and fires FCM sends. No external API calls beyond FCM.
- Add Edge Function logging at entry and exit. If logs are empty after a blood request, the webhook is not reaching the function.

**Warning signs:**
- `net._http_response` table shows `status = 0` or connection errors.
- Edge Function logs are empty after a blood request is posted.
- First-of-day alerts take 3–5x longer than subsequent alerts.

**Phase to address:** FCM push + matching integration phase; webhook delivery must be validated end-to-end with a real blood request before the phase is considered done.

### Pitfall 10: Exact GPS Coordinates Expose Donor Home Address (Privacy / Safety Risk)

**What goes wrong:**
When a blood request is posted, the system stores the requester's precise GPS lat/lon (e.g., `13.7563, 100.5018`). If any API response returns this coordinate to donors — or if a malicious user can query donors' stored locations — the exact home or current address of vulnerable individuals is exposed. In a Myanmar context with political and civil tensions, donor de-anonymization is a serious safety risk, not just a privacy inconvenience. Dating apps have been exploited via trilateration to pinpoint users at home to within meters.

**How to avoid:**
- Store GPS coordinates in the database but never return raw lat/lon to the client. Return only the distance (e.g., "2.3 km away") computed server-side.
- For blood requests: store the precise location but expose only a geohash-reduced area (e.g., ~500m resolution) in any donor-facing UI. Donors need to know "neighborhood" not "exact address."
- Use Supabase RLS to ensure the `location` geography column is never selectable by any role other than the server-side matching function.
- For donors' stored locations (used for matching): these should never be directly queried by other users — the matching query runs server-side and returns only `donor_id` + distance, not coordinates.
- Consider snapping GPS to a 500m grid on write, so even server logs never contain exact home coordinates.

**Warning signs:**
- The API response for a blood request includes `{ lat: ..., lon: ... }` fields visible to any authenticated user.
- The donor list returned from the matching query includes location columns.

**Phase to address:** Database schema + RLS phase (privacy by design at schema creation), and again during API response audit before any public testing.

### Pitfall 11: Burmese Text Rendering Failures — Zawgyi vs. Unicode

**What goes wrong:**
The app renders Burmese text in Unicode (Noto Sans Myanmar). A significant portion of Myanmar users still have Zawgyi-One set as their system font. On these devices, Unicode Burmese text displays as garbled squares or wrong characters because Zawgyi uses a non-standard code point mapping that overlaps with Unicode Myanmar ranges. The user sees broken text and cannot use the app.

**Why it happens:**
Myanmar has two incompatible Burmese font encoding systems: standard Unicode and Zawgyi-One (a legacy encoding that is not Unicode-compliant but remains widely used on older Android devices). A web font like `Noto Sans Myanmar` is Unicode-compliant but cannot fix Zawgyi-encoded input text.

**How to avoid:**
- Ship the app in Unicode only. Use `font-family: 'Noto Sans Myanmar', sans-serif` and explicitly load the Noto Sans Myanmar web font — do not rely on the OS fallback font stack, which may resolve to Zawgyi on older devices.
- Use Google's `myanmar-tools` library (JavaScript, available on npm) to detect if user-entered text is Zawgyi and convert it to Unicode before storing. Input fields for Burmese text should run this detection on submit.
- For UI text (labels, buttons, instructions): all static copy should be stored in Unicode. Do not accept user-pasted Zawgyi for UI labels.
- Test on a physical low-end Android device (or emulator) with Zawgyi as the system font to verify Noto Sans Myanmar overrides correctly.

**Warning signs:**
- Burmese text appears as boxes, wrong glyphs, or is missing stacking diacritics.
- Users report they "can't read" the Burmese text.
- The virama character (U+1039) renders as two separate visible characters instead of creating a stacked consonant.

**Phase to address:** Localization / UI phase. Noto Sans Myanmar must be loaded and tested as a web font (not OS fallback) from day one.

### Pitfall 12: Myanmar Phone Number Format Inconsistency

**What goes wrong:**
The app stores phone numbers as raw user input. Myanmar phone numbers have multiple valid formats: `09xxxxxxxx`, `+959xxxxxxxx`, `9xxxxxxxx`, `01xxxxxxxx` (Yangon landline). If stored inconsistently, the same donor registers twice (once as `09123456789`, once as `+959123456789`), creating duplicate profiles, split FCM tokens, and duplicate alert sends. The "call back" feature also breaks if the stored number is not dialable directly from a `tel:` URI.

**How to avoid:**
- Normalize all phone numbers to E.164 format (`+959xxxxxxxx`) on input, before storing. Use a library like `libphonenumber-js` with the `MM` country code default.
- The `profiles.phone` column should have a UNIQUE constraint on the normalized form.
- The `tel:` href for the callback button must use the E.164 form (`tel:+959xxxxxxxx`).
- Display format can be localized (`09xxxxxxxx`) but storage format is always E.164.

**Warning signs:**
- The same person can create two accounts by entering `09...` vs `+959...`.
- The "Call Donor" button fails on some devices.
- FCM sends go to multiple tokens for what appears to be the same user.

**Phase to address:** Auth / profile phase, at the point phone number is first captured.

### Pitfall 13: Fulfilled Requests Still Alerting Donors

**What goes wrong:**
A blood request is marked "fulfilled" by the requester, but in-flight FCM messages (already queued or delayed) continue to deliver to donors. Worse, the matching query does not filter out fulfilled requests, so a subsequent donor who opens the app sees an old request that has been met and calls a requester who no longer needs blood. In a system where donor trust is fragile, false alarms erode willingness to respond.

**Why it happens:**
FCM does not support message cancellation after send. If the matching query and FCM send happen in the same Edge Function call, all matching donors are alerted at once regardless of whether the request is closed between sends. There is no transactional lock between "request is fulfilled" and "stop new alerts."

**How to avoid:**
- Add a `status` column (`open`, `fulfilled`, `expired`) to `blood_requests` with an index. The matching query must include `WHERE status = 'open'` and this filter must run inside the same transaction that reads the request row.
- Before sending each FCM message batch, re-check request status. If the request was fulfilled between the query and the send, abort.
- Set a TTL on blood requests (e.g., auto-expire after 6 hours) so stale requests cannot linger.
- When a donor opens an alert deep-link, fetch the request status live before showing the "Call" button — display "This request has been fulfilled" if status is no longer `open`.

**Warning signs:**
- Donors report calling a number for blood that was already obtained.
- `blood_requests` table has rows with `status = 'fulfilled'` but the matching query has no status filter.

**Phase to address:** Matching + alert phase; the status filter belongs in the initial matching query design, not as a follow-up patch.

### Pitfall 14: Abuse — Fake Requests Spamming All Nearby Donors

**What goes wrong:**
Because auth is a dummy (no real phone verification), anyone can create a blood request with a fake phone number and spam all nearby donors with push notifications. A single bad actor — or a script — could flood the system with fake B− requests, causing donor fatigue, eroding trust in the platform, and potentially making real emergencies indistinguishable from spam.

**Why it happens:**
The dummy OTP flow requires no real identity proof. Without rate limiting or request throttling, the API accepts unlimited `INSERT` into `blood_requests` from any authenticated session.

**How to avoid:**
- Rate limit blood request creation at the database level with a Postgres policy: a user may not create more than 1 open blood request at a time, enforced via RLS (`WITH CHECK (NOT EXISTS (SELECT 1 FROM blood_requests WHERE user_id = auth.uid() AND status = 'open'))`).
- Add a minimum time between requests: a new request from the same user may not be created within N hours of their last request (enforced via a database constraint or RLS check on `created_at`).
- The "mark fulfilled" action should close the request before a new one can be created.
- Log all requests with user_id and timestamp for post-hoc abuse review.
- In a later phase: consider adding a lightweight CAPTCHA or rate limit by IP at the Edge Function level.

**Warning signs:**
- A single `user_id` has multiple open `blood_requests` rows simultaneously.
- Donors report receiving repeated alerts for similar blood types from the same area within minutes.

**Phase to address:** Blood request creation phase; the "one open request per user" RLS check must be written alongside the INSERT policy.

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Skip token refresh listener (`onTokenRefresh`) | Simpler code at launch | ~10% of active donors silently stop receiving alerts per month; no error visible | Never for this use case |
| Store raw lat/lon in API responses | Simpler distance display logic | Exact home addresses exposed; trilateration risk; trust breach | Never |
| Use `service_role` key in dev browser client | Bypass RLS friction during development | If accidentally shipped, root database access in the browser | Never — use a `supabase.auth.signIn()` test user instead |
| Disable RLS during development | Faster iteration | Easy to forget to re-enable; table ships unprotected | Only acceptable in a local-only Supabase instance with no real data |
| Separate `firebase-messaging-sw.js` alongside vite-plugin-pwa SW | Follows Firebase quickstart docs literally | Scope collision → infinite reload loops after deploy | Never — merge into one SW from the start |
| Store Burmese text as Zawgyi in the database | Works for current users | Cannot search, sort, or display correctly in any standard tool | Never — normalize to Unicode on input |
| Skip the fulfilled-request status check in matching query | Slightly simpler query | Donors receive alerts for already-met requests; trust erosion | Never |
| Blood request with no expiry TTL | Simpler schema | Stale requests match indefinitely, donors alerted for days-old emergencies | Never in production |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| FCM + iOS PWA | Call `getToken()` before the PWA is in standalone mode | Gate `getToken()` on `display-mode: standalone` check; show install prompt otherwise |
| FCM + vite-plugin-pwa | Register `firebase-messaging-sw.js` as a second, separate service worker | Merge FCM init into the single Workbox SW using `injectManifest` strategy |
| FCM foreground messages | Rely on FCM to auto-show notifications when app is open | Add explicit `onMessage()` handler that calls `new Notification()` or shows in-app alert |
| Supabase + `pg_net` webhooks | Assume the webhook fires if the function is saved | Check `net._http_response` table; `pg_net` fails silently on auth/network errors |
| Supabase RLS | Test policies using the Dashboard or service_role key | Test with a real user JWT (`anon` key + signed-in user); service_role bypasses RLS |
| PostGIS ST_DWithin | Use `CREATE INDEX` without `USING GIST` | Always `CREATE INDEX ... USING GIST (location)` and verify with `EXPLAIN ANALYZE` |
| FCM token storage | Store token once at registration and never update | Subscribe to `onTokenRefresh()`, upsert on change, delete on FCM send error |
| Myanmar phone input | Store as entered by user | Normalize to E.164 via `libphonenumber-js` with `MM` default country before any storage |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| No GiST index on donor location | Matching query time grows linearly with donor count | `CREATE INDEX USING GIST` at migration time; verify with `EXPLAIN ANALYZE` | Noticeable at ~5,000 donors; severe at 50,000+ |
| Matching query returns all columns including phone numbers | Over-fetching; phone data in memory of any caller | SELECT only needed columns (`donor_id`, distance) in the matching query | Day 1 — it is a security issue before a performance issue |
| Edge Function cold start in alert critical path | First alert of the day takes 2–5s; subsequent alerts are fast | Keep-warm pg_cron ping; measure P99 not just P50 alert delivery time | At low traffic (ironically) — busy systems stay warm |
| FCM sending to all donors serially in Edge Function | Alert delivery time grows linearly with matching donor count | Use FCM `sendEach()` or batch send to up to 500 tokens per call | More than ~20 matching donors in a dense city area |
| No `status = 'open'` filter in matching query | Every new request re-scans all historic requests | Index on `(status, created_at)` and filter at query time | Immediately after first request is fulfilled |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| Phone numbers selectable via anon key | Harvesting of all donor phone numbers; harassment; spam | RLS policy: phone visible only to the owning user and server-side matching function |
| Exact GPS coordinates in API responses | Home address exposure; trilateration attack; safety risk for donors | Return distance only; never return raw coordinates to client |
| Dummy auth creating unstable `auth.uid()` | RLS `user_id` checks fail; duplicate profiles; split FCM tokens | Use email-derived Supabase Auth user for stable UID across sessions |
| `service_role` key in browser bundle | Root database access for anyone with DevTools | Never instantiate browser Supabase client with service_role; use anon key only |
| No rate limit on blood request creation | One actor spams all nearby donors repeatedly | RLS `WITH CHECK` limiting to 1 open request per user |
| FCM token stored without RLS | Any authenticated user can query all push tokens | FCM token table: user can only read/write their own rows |
| Blood request phone number visible before donor confirms response | Pre-response PII exposure | Reveal phone number only after donor marks "I'm responding," enforced server-side |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| Notification permission prompt on first page load | Users dismiss; iOS requires gesture; trust not established | Explain value first ("You'll get alerted when your blood type is needed nearby"), then prompt after explicit "Enable alerts" tap |
| No explanation that iOS requires home-screen install before push works | iOS donors register but never receive alerts; they think the app is broken | Show a prominent "Install to Home Screen to receive alerts" prompt with step-by-step Safari instructions |
| Burmese text rendered via OS fallback font (Zawgyi device) | Garbled or missing text; app unusable for Zawgyi-system users | Load Noto Sans Myanmar as web font explicitly; detect and convert Zawgyi input |
| Push alert payload shows exact blood request location | Privacy exposure in notification shade | Show "Someone nearby needs B− blood" without GPS coordinates in the notification text |
| "Call" button shown for a fulfilled request | Donor calls a number that no longer needs blood; donor trust damaged | Re-fetch request status on alert open; show "Fulfilled" state if closed |
| No offline state handling on a poor-connectivity network | App shows blank screen or spinner indefinitely in 2G/3G conditions | Cache app shell via Workbox; show offline indicator; queue request submission for reconnection |

## "Looks Done But Isn't" Checklist

- [ ] **iOS push:** FCM token subscription works on desktop Chrome — verify the same flow works after "Add to Home Screen" on an iOS device, not just in a Safari tab.
- [ ] **Service worker:** No reload loop after a new deploy — check Chrome DevTools → Application → Service Workers shows exactly one SW at scope `/`.
- [ ] **Blood compatibility matrix:** Same-type matching passes in tests — also test O− donor → AB+ request (should match), AB+ donor → O− request (should NOT match).
- [ ] **RLS on phone numbers:** Phone column is not returned in a `SELECT *` with an anon-key client authenticated as a different user.
- [ ] **PostGIS index:** `EXPLAIN ANALYZE` on the matching query shows "Index Scan using idx_donor_location," not "Seq Scan."
- [ ] **Stale token handling:** Edge Function logs show a DELETE from `fcm_tokens` when FCM returns `registration-token-not-registered`.
- [ ] **Fulfilled request filter:** Posting a new blood request after marking the previous one fulfilled does not re-alert the same donor pool via in-flight messages.
- [ ] **Foreground notification:** Open the app as a donor, post a matching blood request from another session, and confirm the donor sees a visible alert (not just a console log).
- [ ] **Burmese rendering:** View the app on an Android device with Zawgyi as system font; Noto Sans Myanmar web font overrides correctly and text is legible.
- [ ] **GPS privacy:** No `lat`/`lon` fields appear in any network response visible to a non-owning client.

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| iOS push never worked (SW architecture wrong) | HIGH | Rebuild SW as single merged file; all existing iOS subscribers must re-register tokens |
| RLS leak discovered post-launch with real user data | HIGH | Immediately enable RLS + deny-all policy; audit logs for unauthorized access; notify affected users |
| Blood compatibility direction inverted | HIGH | Fix matrix and SQL; re-test all 64 combinations; notify any users who received wrong-direction alerts during the window |
| Stale token accumulation (donors not receiving alerts) | MEDIUM | Run cleanup query deleting tokens not refreshed in 90 days; add `onTokenRefresh` listener in next deploy |
| Service worker scope collision causing reload loop | MEDIUM | Deploy merged SW immediately; users may need to hard-refresh or reinstall PWA |
| GiST index missing (slow queries) | LOW | `CREATE INDEX CONCURRENTLY` (no downtime) then verify with `EXPLAIN ANALYZE` |
| Dummy auth creating duplicate users | MEDIUM | Deduplicate profiles by phone; merge FCM tokens; add UNIQUE constraint on normalized phone |

| Pitfall | Prevention Phase | Verification |
|---------|------------------|--------------|
| iOS push requires home-screen install first | PWA scaffolding + push subscription | Test on physical iOS device in Safari: token only obtained in standalone mode |

| RLS leaking phone numbers | Database schema (first migration) | `SELECT phone FROM profiles` with anon key + non-owner JWT returns 0 rows |
| Blood compatibility direction | Matching logic (before any SQL written) | Unit tests cover all 8 donor types × 8 recipient types; inverted cases explicitly tested |
| Dummy auth leaking to RLS | Auth phase (first phase) | Same phone number produces same `auth.uid()` across sessions; no service_role in browser |
| Missing GiST index | Database schema (same migration as location column) | `EXPLAIN ANALYZE` shows Index Scan |
| Edge Function cold start / pg_net silent failure | Matching + push integration | `net._http_response` shows 200; Edge Function logs show execution on every blood request insert |
| Exact GPS in API responses | Database schema + RLS (privacy review) | Network tab shows no lat/lon fields in any client-visible response |
| Burmese Zawgyi rendering | UI / localization phase | Tested on Zawgyi-system Android device; Noto Sans Myanmar loads and displays correctly |
| Phone number format inconsistency | Auth / profile phase | UNIQUE constraint on normalized E.164 phone; duplicate registration blocked |
| Fulfilled request still alerting | Matching + alert phase | Marking request fulfilled stops alert sends; donor sees "fulfilled" state on alert open |
