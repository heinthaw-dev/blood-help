<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → donations Realtime channel | A donor client subscribes to donation events; must only receive its own |
| requester client → blood_requests UPDATE (close/extend) | Owner-supplied writes to status/closed_at/expires_at/extended cross into the row |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-03-01 | Information Disclosure | A client subscribing to another donor's donations | mitigate | Channel filter `donor_id=eq.${uid}` + donations SELECT RLS scoped to the two parties (09-01); Realtime honors RLS so foreign rows never arrive |
| T-09-03-02 | Elevation of Privilege | A user closing/extending another user's request | mitigate | Owner-scoped writes `.eq('requester_id', uid)` and the existing owner UPDATE RLS policy on blood_requests (verified in 09-01) |
| T-09-03-03 | Tampering | Extending a request more than once | mitigate | `handleExtend` only fires when `!activeRequestExtended`; the `extended` flag is set on the write and hydrated from DB (Pitfall 5), so a second extend is blocked client-side and the banner stays hidden |
| T-09-03-04 | Information Disclosure | Stale `lastSeenDonationAt` leaking a prior user's congrats on a shared device | mitigate | `handleLogout` clears the marker (Task 1) |
| T-09-03-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this plan (react-zxing installed in 09-02) |
</threat_model>
