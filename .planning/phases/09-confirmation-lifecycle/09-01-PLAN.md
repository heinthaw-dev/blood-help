<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| authenticated client → confirm_donation RPC | Untrusted requester input (request_id, donor_code) crosses into a privileged SECURITY DEFINER function |
| pg_cron (in-DB) → blood_requests | Trusted in-DB scheduled writer; no client involvement |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-01 | Elevation of Privilege | confirm_donation RPC called on another user's request | mitigate | Step-1 ownership gate `auth.uid() = requester_id`; non-owner returns generic `invalid_code` (Task 2) |
| T-09-02 | Spoofing | Confirming a non-participating donor by guessing codes | mitigate | Participant check requires a `request_responses` row with `status='responding'` (D-04); generic `invalid_code` hides whether a code is real (D-06) |
| T-09-03 | Tampering | Double-crediting the same donor (replay) | mitigate | Unique `(request_id, donor_id)` constraint + explicit duplicate check returning `already_confirmed` (D-09) |
| T-09-04 | Information Disclosure | Loosening donations RLS to enable Realtime would leak donations to non-parties | mitigate | REPLICA IDENTITY FULL + publication membership only; SELECT RLS stays scoped to the two parties (publication membership does not bypass RLS) |
| T-09-05 | Tampering | Direct client UPDATE of blood_requests.status bypassing business logic | accept | Status transitions to fulfilled/expired happen only via the RPC or pg_cron; owner close-writes (09-03) are an owner-permitted path; table UPDATE RLS already scopes to owner |
| T-09-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs in this plan (react-zxing install is in 09-02, covered there); RESEARCH §"Package Legitimacy Audit" rates react-zxing [OK] |
</threat_model>
