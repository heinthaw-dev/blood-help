<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client (anon key) → Postgres | Public anon key reaches Supabase; RLS is the only access-control layer |
| authenticated user → donors/blood_requests tables | A user must only access their own rows; other users' donor contact details must not be exposed |
| client → SECURITY DEFINER RPC | RPCs bypass RLS by design; they must return only feed-safe columns |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Elevation of Privilege | INSERT/UPDATE another user's donors row | mitigate | donors_insert_own / donors_update_own RLS WITH CHECK `profile_id = (SELECT auth.uid())` (spec §4.3) |
| T-07-02 | Information Disclosure | Reading other users' raw donors rows | mitigate | donors_select_own RLS USING `profile_id = (SELECT auth.uid())`; no SELECT-others policy; anon has no access |
| T-07-03 | Information Disclosure | requests_within_radius SECURITY DEFINER leaking more than the feed needs | mitigate | RETURNS TABLE lists only active-request feed columns (no donor PII); contact_phone is the REQUESTER's number, intentionally shown to donors per spec §3.4; returns only status='active' AND expires_at>now() rows |
| T-07-04 | Information Disclosure | donors_within_radius leaking donor phone | mitigate | RPC returns only id, profile_id, blood_type, donation_count, lat, lng, dist_meters — no phone/contact column |
| T-07-05 | Spoofing | donor_code enumeration (Phase 9 concern) | accept | 32^5 ≈ 33M space, UNIQUE constraint, not exposed in Phase 7 — low risk now |
| T-07-SC | Tampering | npm/pip/cargo installs | accept | No new packages this phase (07-RESEARCH §Package Legitimacy Audit — none added); no install task present |
</threat_model>
