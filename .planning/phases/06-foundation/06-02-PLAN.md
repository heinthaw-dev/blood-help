Enable Row-Level Security on all 5 tables and deploy all RLS policies via Supabase MCP apply_migration. Policies implement the spec §4.3 access rules: own-profile full access, active requests visible to all authenticated users, responses/donations visible to the two parties only.

Purpose: Without RLS, any authenticated user can read any row. This plan locks down access at the database layer — the authoritative enforcement point that cannot be bypassed from the React client.
Output: All 5 tables have RLS enabled and correct policies in place, verifiable in the Supabase dashboard under Authentication > Policies.

    .planning/phases/06-foundation/06-RESEARCH.md Pitfall 3 (column-level security does not exist in Postgres — phone hiding requires own-row-only SELECT)
  </read_first>
  <action>
    Use the Supabase MCP apply_migration tool to run migration named "rls_profiles_requests_tokens".

    For each table listed below, first enable RLS then create all policies. Use `to authenticated` on all policies (per Pitfall 2: anonymous auth users have the `authenticated` role, not `anon`). Use the performance-safe form `(select auth.uid())` rather than `auth.uid()` directly in USING/WITH CHECK clauses.

    PROFILES table:
    - `alter table public.profiles enable row level security;`
    - SELECT policy "own_profile_select": USING `(select auth.uid()) = id` — own row only. This also prevents phone column exposure to other users (per Pitfall 3: no column-level RLS; restricting to own row is the correct approach for Phase 6).
    - INSERT policy "own_profile_insert": WITH CHECK `(select auth.uid()) = id`
    - UPDATE policy "own_profile_update": USING `(select auth.uid()) = id` WITH CHECK `(select auth.uid()) = id`

    BLOOD_REQUESTS table:
    - `alter table public.blood_requests enable row level security;`
    - SELECT policy "active_requests_select": USING `status = 'active'` — all authenticated users can see active requests. (contact_phone field is accessible to all authenticated users in Phase 6; Phase 7 geo-match RPC will gate contact_phone to within-range donors per spec §4.3 and RESEARCH.md Pitfall 3 / Open Question 2. This is the accepted Phase 6 posture — the verifier should not flag contact_phone access as a Phase 6 gap; it is explicitly deferred to Phase 7.)
    - INSERT policy "requester_insert": WITH CHECK `(select auth.uid()) = requester_id`
    - UPDATE policy "requester_update": USING `(select auth.uid()) = requester_id` WITH CHECK `(select auth.uid()) = requester_id`

    DEVICE_TOKENS table:
    - `alter table public.device_tokens enable row level security;`
    - ALL policy "own_tokens_all": USING `(select auth.uid()) = profile_id` WITH CHECK `(select auth.uid()) = profile_id`
  </action>
  <verify>

    - SELECT policy "response_parties_select": USING clause allows read when `(select auth.uid()) = donor_id` OR when `(select auth.uid())` is the requester_id of the linked blood_request:
      `(select auth.uid()) = donor_id or (select auth.uid()) in (select requester_id from public.blood_requests where id = request_id)`

    - `alter table public.donations enable row level security;`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Supabase API → Postgres | All client queries pass through Supabase's JWT verification before RLS evaluation |
| React client → Supabase | React uses anon key; actual access gated by RLS using the JWT's auth.uid() |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-04 | Information Disclosure | profiles.phone column | mitigate | Own-row-only SELECT policy; no other user can SELECT another user's profile row including phone |
| T-06-05 | Information Disclosure | blood_requests.contact_phone | accept | Phase 6: simple active-requests SELECT allows all authenticated users; Phase 7 geo-match RPC will restrict to in-range donors per spec §4.3 — this is the accepted Phase 6 posture; contact_phone gating to within-range donors is deferred to Phase 7 and is NOT a Phase 6 gap |
| T-06-06 | Elevation of Privilege | request_responses INSERT | mitigate | WITH CHECK `auth.uid() = donor_id` prevents inserting responses as another donor |
| T-06-07 | Information Disclosure | donations visibility | mitigate | SELECT policy restricts to donor_id and recipient_id only |
| T-06-08 | Spoofing | anon vs authenticated role confusion | mitigate | All policies use `to authenticated`; anonymous auth users hold authenticated role per Supabase docs (Pitfall 2) |
| T-06-SC | Tampering | npm/pip/cargo installs | accept | No package installs in Wave 1 — schema-only via MCP |
</threat_model>
