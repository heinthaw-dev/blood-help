  <action>
    Use the Supabase MCP apply_migration tool to run migration named "donors_within_radius_rpc".

    Create the function `public.donors_within_radius` with these exact properties:
    - Parameters: `lat double precision, lng double precision, radius_km double precision`
    - Returns: TABLE with columns: id uuid, name text, blood_type blood_type, donation_count int, lat double precision, lng double precision, dist_meters double precision
    - `set search_path = ''` — required for PostGIS extensions. prefix to work correctly
    - `language sql`
    - `security definer` — so the function can read profiles table bypassing per-row RLS for the geo-distance query (callers see only returned columns, not raw profile rows)
    - Body: SELECT from `public.profiles p` WHERE `p.is_donor = true AND p.is_available = true` AND `extensions.st_dwithin(extensions.st_point(p.lng, p.lat)::extensions.geography, extensions.st_point(lng, lat)::extensions.geography, radius_km * 1000)` — note the * 1000 for meters (Pitfall 4)
    - Include `dist_meters` as `extensions.st_distance(extensions.st_point(p.lng, p.lat)::extensions.geography, extensions.st_point(lng, lat)::extensions.geography)` in the SELECT list
    - ORDER BY dist_meters

    After the CREATE OR REPLACE: grant execute on the function to the `authenticated` role:
    `grant execute on function public.donors_within_radius(double precision, double precision, double precision) to authenticated;`

    Note on parameter naming: the function's local parameters `lat` and `lng` shadow the column names in `public.profiles`. Alias the table as `p` (already in pattern) and use `p.lat`/`p.lng` for the column references and `lat`/`lng` for the parameter references to avoid ambiguity.
  </action>
